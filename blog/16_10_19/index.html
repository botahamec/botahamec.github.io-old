What's Webster Supposed To Be?

Webster is supposed to be an AI, in a sense, which keeps track of objects. It's split into two parts, a dictionary, and a memeory. The ditionary contains definitions of all possible objects, and the memory contains descriptions of all the objects which it knows to exist.

"But Bota, this just sounds like a normal database." Well, here's the catch, based on what it knows already, it make make inferences about objects in the world. For example, since this thing is round and red, there's a good chance it's an apple, and because it's an apple, it probably tastes sweet.

Ok, So What's Wrong With That?

Here's the thing: I was hoping I could make Webster simple. Boy, was that a mistake. Because of its simplicity, it was simply impractical to do any of this advanced inferencing. One line of logic I was planning on using was, for example, "This thing is a ruby. I thought rubies were green. It turns out they are red. This thing is red." There's an obvious problem with that. It would be more reasonable in this context to assume that the object is not a ruby. The problem, in the end, was that the objects didn't contain enough information.

Another problem is that the definitions are too simplistic. For example. The background of this page is purple (At least, it was when I wrote but it might have changed). So let's define the color purple. Purple is a color with a hex value of #8800ff. Here's the problem: the hex value for the page is not #8800ff (It's actually #2f0345). But I think we can still agree that it's purple. While we agree with this broad definition of purple, Webster has one idea of what purple is and it's not willing to accept any alternatives.

One final problem: scale. I want Webster to eventually have a dictionary comparable to the produced by the collaboration of Mariam and Webster. Meaning it should contain just about every word in the English language. I do not plan on rerwriting the entire dictionary into Webster, and neither are you. So now we have a new problem, which is getting it to learn.

Oh Noes! What Do We Do?

As you can tell by my master's thesis of a docstring above, I already have some solutions. I'll explain the solution I gave in the docstring first. More complicated Definition and Object classes. I won't get into too many specifics, but the main idea is that instead of defining a value for each property of a definition, I would define rules. So eventually it would boil down to a boolean expression which would say, definitively, if the object is of the given definition.

Another thing which needs to be done is for the Object class to have a "definition" property. This way, there can be some ambiguity to what the object actually is, but it can be definitive if we know for sure that it has the given definition. I don't have a great way of explaining it, but trust me, my ruby example from before works under this system.

The last problem I illustrated is less solvable, but I'll figure that part out later. I'm sure it's doable, but I think the solution is outside the scope of this article.

But Wait! I Have More Planned

So far I've been focusing exclusively on nouns, but I also want to implement other types of words (there's a word that people who know stuff about language would use here, but as you can see I don't know anything about language), such as adjectives and verbs. Adjectives would just have a boolean value representing whether or not the given adjective describes it. Verbs are more complicated. I'll figure that out later.

I also want to have a sort of "digital makefile", which would allow an object to determine some of its properties based on its other properties. For example, based on the length of each side of a triangle, I can easily figure out what its area is, possibly using the laws of sines and cosines.

And if it could have a shell, that'd be great. Just type values and definitions and other things into the shell and it learns.
